AiBICI Website

[ Django + React + mariaDB ] 구성할 계획이었으니 선배 프로그래머 입사함에 따라 [ express + React + mariaDB ] 변경함

express + Prisma 를 이용하여 mariaDB를 제어하며 프론트앤드에서 효율적으로 원하는 정보를 가져 갈 수 있도록 REST 에서 GRAPHQL 로 변경함

프로젝트를 통해 배운 내용(진행중)

1. 요구사항정의서
- 정확한 요구사항이 무엇인지를 확인해야 하며 사용자의 모든 행위(제한해야 하는 행위 포함)를 이해하고 있어야 한다.
- 에피소드 :
a. 관리자의 승인을 받는 서비스에 대하여 원본을 수정 할 경우 계속적으로 승인을 받을지와 사본을 만들 경우 상황에 따라 원본과 사본을 구분하여 보여주는 경우

2. 데이터베이스 설계
- 각 역할별로 DB를 분리하며 table은 중복되지 않도록 작성하여 원하는 정보에 쉽고 빠르게 접근 할 수 있도록 해야 한다.
- 에피소드 :
a. 서비스를 제공하는 id는 고정을 하되 그 내용을 변경할 수 있도록 기능을 추가해야 함에 DB 구조 변경 및 이전
b. 최초의 요구사항으로 DB를 구성하였지만 변경된 요구사항으로 인하여 추가되는 기능을 DB에 적용

3. REST API & GraphQL & JSON ( JavaScript Object Notation )
- REST API의 경우 URL의 Method 와 파라미터를 이용하여 원하는 정보를 제공하며 JSON 형태를 작성하여 행위를 제어할 수 있다.
- REST API의 경우 과도한 정보를 제공하거나 네트워크 자원을 낭비하는 경우가 있기에 그 대안책으로 GraphQL이 있다
- 에피소드 :
a. 정해진 정보만 전달함에 프론트앤드에서 불필요한 정보를 구분해야 하며 제3자가 전달된 정보 중 특정 정보를 분석하여 시스템을 분석할 수 있는 경우
b. 요구사항정의서가 부족함에 전달해야 하는 데이터가 유동적으로 변경해야 함에 GRAPHQL이 효율적이라 판단되어 구조를 개선함

4. JWT ( JSON WEB TOKEN )
- 본인 인증시 Refresh Token을 DB에 저장하고 Access Token을 발행하며 만료시점에 Access Token을 갱신하며, JWT의 Header는 비대칭 암호화를 이용하며 Payload의 정보는 암호화하여 최소한 외부에 정보를 확인 할 수 있도록 한다.
- 에피소드 :
a. 발행된 Access Token과 DB에 저장된 Refresh Token에서 지정된 시간이 만료되기 전에 DB에 저장된 Refresh Token이 동일 할 경우 Access Token을 재발행하는 것을 반대로 이해해서 CODE 작성시 오류가 발생
b. Payload의 정보를 암호화 및 복호화를 적용하여 jwt.io에서 보았을 경우 그 값들을 볼 수 없도록 처리함 (But REST API에서 그 값들을 볼 수 있었음)

5. JAVASCRIPT ( Node.js의 Express )
- 여러가지 함수형태와 객체와 배열, 제어문등 사용 방법과 async와 await를 이용하여 비동기 처리를 하는 것을 조금은 이해한다.
- 에피소드 :
a. 제어문 많은 if보다 switch를 사용하는 것이 더 빠르게 진행 할 수 있으며, 삼항연산자의 경우 중복된 CODE를 줄일 수 있게 함
b. 배열 메서드(slice, filter 등)와 map을 이용하여 for문을 대처할 수 있도록 함

6. Prisma ORM ( Object-relational mapping )
- REST API와 GraphQL을 지원하는 ORM으로 JAVASCRIPT의 형태를 따라가서 빠르게 적용 할 수 있도록 지원을 한다.
- 에피소드 :
a. 동적인 DB 이전을 위해 ORM을 사용하는데 undefined일 경우와 findUnique와 findMany의 적용시 그 범위에 대해 이해하는데 시간이 필요했음
b. Prisma를 통하여 가져온 DB를 바로 전달하지 않고 JAVASCRIPT를 이용하여 원하는 형태로 재가공하여 보내는 경우도 있으며, DB의 CRUD를 JAVASCRIPT로 대처하여 횟수를 줄이는 것이 더 효율적인 경우가 있음

7. CODE 작성법
- 일반적인 문서 작성시 관련된 제3자가 보더라도 이해할 수 있도록 작성해야 하며 CODE 작성 또한 문서 작성과 비슷하다.
- 에피소드:
a. 파일들을 model과 controller로 구분하여 작성하면서 기능이 많아짐에 따라 기능과 관련된 파일들의 위치가 멀어져서 확인이 불편함.
b. 주석을 통하여 CODE 흐름을 빠르게 이해하도록 도움을 줘야 하지만 작성된 주석을 다시 확인 했을 때 이해가 안되는 경우
